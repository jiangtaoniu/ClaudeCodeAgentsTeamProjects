# TensorRT 模型编译与底层优化详解

本篇文档为您详细介绍本项目中 **`03_tensorrt_build`** 模块的设计与实现。该模块负责将通用的 ONNX 模型编译为专属于特定 GPU 显卡硬件架构的 **TensorRT Engine（.engine 文件）**，以实现车载计算平台上的极致推理性能。

---

## 总体编译流程预览

TensorRT 本质上是一个 **“深度学习编译器”**。它不负责模型的训练，而是将代表“计算逻辑蓝图”的 `.onnx` 文件，翻译并固化为能够被目标 GPU 执行的高性能“机器码二进制文件（`.engine` 引擎）”。

```text
       pointpillar.onnx (通用计算图 3D 模型)
              │
              ▼ 【第一步】 Parser 将网络图解构进入内存
       OnnxParser 解析器解析
              │
              ├─► 1. 垂直/水平算子融合 (Layer Fusion)
              ├─► 2. 混合精度转换 (FP16 激活 Tensor Core)
              └─► 3. 硬件自适应测试与核函数选择 (Kernel Auto-tuning)
              │
              ▼ 【第二步】 动态输入维度区间锁定
       Optimization Profile 优化配置文件指派
         - Min: [1, 32, 4]  (空旷场景保底)
         - Opt: [16000, 32, 4] (常规场景极限加速)
         - Max: [40000, 32, 4] (拥堵路口安全上限)
              │
              ▼ 【第三步】 序列化写盘
       Serialized Network Output ────→ pointpillar.engine
```

---

## 一、 第一步：解析计算图 (ONNX Parsing)
```python
parser = trt.OnnxParser(network, TRT_LOGGER)
```
TensorRT 启动其内置的解析器（OnnxParser），遍历 `pointpillar.onnx` 文件。它并不直接关注 Python 层的代码逻辑，而是将模型解析为一系列由数学关系相连的**拓扑节点图**，并读入导出时存储在文件中的训练权重。

---

## 二、 第二步：TensorRT 编译器的核心底层优化
一旦读入网络模型，TensorRT 优化器会针对您当前的 GPU 设备，开展以下三项极为关键的“榨干硬件算力”优化：

### 1. 算子融合 (Layer Fusion)
在普通的 PyTorch 模型层级结构中，每个计算层是完全隔离的。数据从显卡全局显存（Global Memory）移入计算小核心（ALU），计算完成后还必须移出写回显存。
显卡芯片在运行时，**数据频繁的搬运和读写交互（内存带宽）所产生的耗时，往往远胜于乘加矩阵计算本身**。

*   **垂直算子融合**：
    TensorRT 会自动探测网络，把前后的 `Conv`、`BatchNorm` 和 `ReLU` 节点融合成一个名为 **CBR（Conv-BatchNorm-ReLU）** 的超级大算子。数据一次读入 GPU 高速缓存，一次性完成三个计算，仅需写回显存一次，**将原本 3 次的显存交互降低为 1 次**。
*   **水平算子融合**：
    如 Backbone 2D 网络中有着三个平行并列的反卷积（Deconv）分支，它们有着相同的特征输入。TensorRT 会将它们合并为一个宽通道的单卷积节点进行计算，减少对同一块显存的重复访问频率。

### 2. 混合精度计算 (FP16 优化)
```python
if builder.platform_has_fast_fp16:
    config.set_flag(trt.BuilderFlag.FP16)
```
*   **做法**：当探测到显卡支持高速半精度计算时，TensorRT 会将原本 32 位的单精度浮点数（FP32）计算压缩并强制转换为 16 位半精度浮点数（FP16）。
*   **收益**：
    1.  模型运行时，数据在显卡各部件中流动所占用的**总线带宽大小直接削减一半**。
    2.  激活显卡内部专门针对深度学习设计的 **Tensor Core（张量核心）** 并行加速单元，数据处理吞吐量提升数倍。
    3.  *（注：由于 3D 物体检测对极微小的浮点数精度变化不敏感，FP16 方案在此项目下精度的损耗几乎可以忽略不计。）*

### 3. 硬件自适应测试与核函数选择 (Kernel Selection)
对于相同的矩阵卷积公式，在 GPU 的 CUDA 微观驱动层面有着几十种截然不同的代码执行排列方式（如采用不同的共享内存分块机制、不同的线程块对齐策略）。
*   **测速试车**：在您运行 `build_engine.py` 时，TensorRT 编译器会在您的当前显卡上现场运行小规模计算测试。它将每一种计算方案在您的 GPU 上做一次“盲测运行时间评估”，精选出在您目前显卡架构下**跑得最快的物理核函数（Kernel）**，并保留为最终代码。
*   *（注：这也是为什么生成的 `.engine` 文件不能直接跨显卡运行的原因。如果在 RTX 4070 上生成的引擎拿到 RTX 3060 或 Jetson Orin，由于内部物理结构改变，原本最快的核函数可能会由于寄存器限制导致严重降速乃至直接报错崩溃。）*

## 三、 第三步：动态维度优化配置文件 (Optimization Profile)
在 3D 点云检测中，每帧激光雷达扫出的障碍物和点云是不规则变化的，这导致传入模型的非空 Pillar 网络数量（`voxel_num`）大相径庭。

为了让 TensorRT 能在**尺寸动态变化**的前提下，兼顾高效的内存空间分配，我们在代码中引入了配置优化文件：
```python
profile = builder.create_optimization_profile()
profile.set_shape('voxels', (1, 32, 4), (16000, 32, 4), (40000, 32, 4))
profile.set_shape('voxel_num_points', (1,), (16000,), (40000,))
profile.set_shape('voxel_coords', (1, 4), (16000, 4), (40000, 4))
```
这三个维度组合代表了该动态输入轴对应的不同极限参数：
1.  **最小值 Min `[1, 32, 4]`**：极度空旷场景下，哪怕雷达扫过去只有 1 个回波点，也能完全走通计算，绝对不发生死锁或数组越界崩溃。
2.  **最大值 Max `[40000, 32, 4]`**：繁华十字路口下模型可容许的非空格子数安全天花板。如果超出上限，会自动做物理截断，防止显存碎片溢出崩溃。
3.  **优选值 Opt `[16000, 32, 4]`**：平均路况下**最常见、概率最高**的网格密度。TensorRT 会主要针对 16000 规模做最细致的指令排布与数据对齐，确保绝大多数平稳行车路线上能发挥极高效率。

在车端程序实际推理运行时，执行器（Runtime）会根据当前这帧实际的格子数，在这三档中动态地挑选最优的硬件执行代码。

---

## 四、 第四步：序列化导出 (Serialization)
```python
engine_bytes = builder.build_serialized_network(network, config)
with open(TRT_ENGINE, 'wb') as f:
    f.write(engine_bytes)
```
当图优化、混合精度转换、以及自适应硬件测速优化全部执行完毕后，TensorRT 将当前保存在内存里包含了所有特定优化硬件代码、算子融合图、最终权重的模型结构对象**序列化**转换为一个二进制的大数据块（Byte Stream），并最终导出为 `pointpillar.engine` 模型引擎。

该 `.engine` 文件在后续的 ROS 2 推理运行节点中被载入反序列化使用。
